# M2.1 — 신호 루프 검증 설계

> 작성일: 2026-04-23
> 선행: M1 Golden Path (서버 파이프라인 완성, HTTPS E2E 검증됨)
> 메모리: "큰 기능들이 먼저 작동하면, 그다음에 실제 역할 수행"

## 1. 목적

실 Gmail/AdsPower/브라우저 액션 없이, **워커 ↔ 서버 ↔ 대시보드** 세 노드 사이의 신호가 완전히 오가는지 검증한다. 파이프라인의 "통신 골격" 이 흔들림 없이 돌아야 그 위에 실 액션 로직을 올릴 수 있다.

**검증 관점**:
- 대시보드에서 누른 버튼이 서버를 거쳐 워커에 닿는가
- 워커의 heartbeat/fetch/complete 가 서버 상태를 업데이트하고 대시보드가 실시간으로 본다
- 상태 머신 전이가 워커 동작에 따라 흐른다 (자동 전이 진짜 작동)
- 감사/로그 이벤트가 순차 기록된다

실제 Google 로그인, YouTube 댓글 작성 등 "로직" 은 **범위 밖** — M2.2 로 미룸.

## 2. 스코프

### 포함 (M2.1)
- 워커 DRY-RUN 모드 (env `HYDRA_WORKER_DRY_RUN=1`) — executor 가 실 로직 안 타고 즉시 `complete` 리턴
- Mac 로컬 워커 1대 기동 스크립트 완성 (VPS enrollment → worker_token 자동 저장 → `python -m worker` 실행)
- 대시보드/워커/계정 페이지 실시간 갱신 주기 단축 (2~5초)
- 태스크 큐 가시성 — pending/running/done/failed 카운트 + 최근 흐름
- 배정 현황 — 각 워커가 지금 어떤 task 수행 중
- E2E 스모크 확장: 실 워커 접속 상태 + 자동 전이 루프 1회 완주 자동 검증

### 제외 (M2.2 이후)
- 실 Gmail 로그인, 실 AdsPower 프로필 open, 실 YouTube 액션
- 캠페인 생성/편집 UI (스텁 캠페인은 유지)
- 프리셋 편집기
- UI 전면 재설계 ("토스 수준")
- M1-12 concerns (OTP 저장 경로, identity_challenge 전이) — DRY-RUN 에선 불필요, 실 액션 단계 (M2.2) 에서 처리

## 4. DRY-RUN 모드 구현

**파일**: `worker/executor.py` 최상단 또는 execute 메소드 진입부.

```python
# worker/executor.py
import os

_DRY_RUN = os.getenv("HYDRA_WORKER_DRY_RUN", "").lower() in ("1", "true", "yes")

class TaskExecutor:
    async def execute(self, task: dict) -> dict:
        if _DRY_RUN:
            # 실 로직 건너뛰고 바로 성공. 짧은 인공 delay 로 "실행 흔적" 남김.
            await asyncio.sleep(0.5)
            return {"ok": True, "dry_run": True, "task_type": task.get("task_type")}
        # 기존 실 로직 그대로
        ...
```

**설계 원칙**:
- 환경변수 ON → 즉시 성공 (task_type 무관)
- 0.5초 sleep — 대시보드에서 running 상태를 관찰 가능하게 하는 의도
- 기존 실 로직 분기 유지 — 환경변수 OFF (기본) 이면 원래대로

## 5. Mac 로컬 워커 실행 스크립트

**파일**: `scripts/run_mac_worker_dry.sh`

역할:
1. `.env` 에서 SERVER_URL / ADMIN_EMAIL / ADMIN_PASSWORD 로드
2. `/api/admin/auth/login` → admin token
3. `/api/admin/workers/enroll` → enrollment_token
4. `/api/workers/enroll` → worker_token + secrets
5. `.env` 에 `WORKER_TOKEN`, `HYDRA_WORKER_DRY_RUN=1` 주입
6. `python -m worker` foreground 실행

출력: heartbeat 수신 / task fetch / dry-run complete 로그가 실시간으로 터미널에 나옴.

## 6. 실시간 대시보드 갱신

**기존 상태**:
- `/` 대시보드: setInterval 30s (stats + workers + campaigns)
- `/accounts`: setInterval 10s
- `/workers`: setInterval 없음 (페이지 진입 시 1회)
- `/tasks`: 확인 필요

**변경**:

| 페이지 | 기존 | M2.1 | 대상 API |
|---|---|---|---|
| `/` 대시보드 | 30s | 10s | 기존 /api/stats, /api/workers |
| `/accounts` | 10s | 3s | /accounts/api/list |
| `/workers` | 진입 시 1회 | 3s polling 추가 | /api/admin/workers/ + /api/workers/ |
| `/tasks` | ? | 5s | 신규 /api/admin/tasks/stats |

주기는 조정 가능 (너무 짧으면 서버 부담). polling 으로 충분하며, WebSocket 은 M2.4 UI 재설계 단계에서 고려.

## 8. 워커 배정 현황

`/workers` 페이지 카드에 **현재 실행 중 task ID + task_type** 표시. 데이터 소스:
- `GET /api/admin/workers/` 응답에 **running task 요약** 추가
- 또는 `/api/admin/tasks/recent` 와 클라이언트에서 매핑

구현은 단순한 쪽 — `/api/admin/workers/` 응답에 `current_task: {id, task_type, started_at}` 필드 추가 (pytest 로 검증 가능).

## 9. E2E 스모크 확장

**`scripts/e2e_check.sh`** 에 추가:
- `/api/admin/tasks/stats` 접근 확인
- `/api/admin/tasks/recent` 접근 확인
- 배정 현황 API 응답 구조 확인

**신규 스크립트**: `scripts/m2_1_loop_check.sh`
- 테스트 계정 1개 `register` 호출
- Mac 로컬 워커가 DRY-RUN 으로 돌고 있다는 전제
- 20초 대기 → Account status 가 warmup(day1) 이상으로 전이됐는지 확인
- 30초마다 polling → warmup 완주 (day 4 / active) 도달 확인
- 60초 내 도달 못 하면 실패 (DRY-RUN 축소 모드라 빠른 응답 기대)

## 10. 테스트 전략

**단위**:
- `_DRY_RUN` 환경변수 플래그 정확히 해석 (1/true/yes 대소문자 허용)
- `TaskExecutor.execute` 가 DRY-RUN 시 실 로직 import 안 함 (mock 으로 hydra.* 호출 카운트 0 검증)
- 신규 `/api/admin/tasks/stats` 응답 구조 (pytest TestClient)
- 신규 `/api/admin/tasks/recent` 응답 구조

**통합**:
- 가짜 워커 (TestClient) 가 fetch → complete 루프 돌릴 때 대시보드 stats 업데이트 확인

**E2E (수동 포함)**:
- Mac 로컬 워커 기동 → 브라우저 `/workers` 에서 "online" 실시간 반영
- `/accounts/register` 클릭 → `/tasks` 에서 pending → running → done 순서 관찰
- pause 누르면 워커 fetch 멈춤 → `/tasks` 에서 pending 정체
- unpause 복원

## 11. 실패/복구

| 시나리오 | 처리 |
|---|---|
| Mac 워커 네트워크 단절 | heartbeat 실패 → 15초 후 재시도 (기존 로직) · /workers 에서 "오프라인" 표시 |
| DRY-RUN 태스크 실패 시뮬레이션 | `HYDRA_WORKER_DRY_RUN_FAIL_RATE=0.2` env 로 20% fail 유도 (선택 기능) — 재시도/suspended 흐름 검증 |
| 실 로직 사고 방지 | DRY-RUN 모드에서 실 HTTP 호출 (`/api/generate-comment`, AdsPower, Google) 발생 안 하도록 방어. lazy import 로 충분 |

## 12. 완료 기준

1. `pytest` 전체 GREEN
2. Mac 로컬 워커가 `HYDRA_WORKER_DRY_RUN=1` 로 기동 → VPS heartbeat 수신 확인
3. 어드민 UI `/workers` 에 "online" 표시 · 3초 내 갱신
4. `/accounts` 에서 계정 1개 등록 → **45초 이내** 에 `warmup → active` 까지 DB 상태 자동 전이 확인 (orchestrator 이벤트 훅 + DRY-RUN 속도)
5. `/tasks` 페이지에서 pending → running → done 카운트 변화 관찰
6. pause 클릭 → 워커 fetch 중단 (heartbeat/v2 응답의 paused 반영)
7. unpause 클릭 → 복귀
8. `scripts/m2_1_loop_check.sh` 자동 스모크 PASS
