# HYDRA 어드민 재설계 — 클로드 코드 작업 문서

이 디렉토리는 HYDRA 어드민 UI/UX 재설계 작업을 클로드 코드에게 단계적으로 시키기 위한 문서 모음입니다.

---

## 사용 방법

### 첫 작업 시작할 때

클로드 코드 세션을 열고 다음 두 파일을 읽게 합니다:

1. `CLAUDE.md` — 프로젝트 비전, 디자인 원칙, 기술 스택, 코딩 컨벤션 (모든 작업 공통)
2. 작업할 PR 문서 1개 (예: `PR-1-terminology.md`)

추천 시작 명령:
```
@CLAUDE.md @PR-1-terminology.md

위 두 문서를 읽고 PR-1 작업을 시작해줘.
완료 정의를 모두 만족할 때까지 진행하고,
중간에 막히면 결정이 필요한 부분 명확히 알려줘.
```

### 다음 PR로 넘어갈 때

이전 PR이 머지되고 안정화 확인된 후 다음 PR 시작:
```
@CLAUDE.md @PR-2-pipeline-dashboard.md

PR-1 머지된 상태야. PR-2 시작해줘.
```

## 각 PR 작업 시 클로드 코드에게 강조할 것

이 문서들은 의도와 제약을 다 적었지만, 클로드 코드는 가끔 큰 그림을 놓칠 수 있어요. 다음을 매번 짚어주세요:

### 시작할 때
- "CLAUDE.md의 디자인 원칙 5개를 항상 지켜라"
- "운영 중 시스템이라 데이터 날리면 안 됨"
- "완료 정의를 모두 만족시키지 못하면 PR 안 올림"

### 코드 리뷰할 때
- "용어 매핑이 i18n-terms.ts로 통일됐는가"
- "API 응답 schema가 Zod로 검증되는가"
- "빈 상태 / 로딩 / 에러 모두 처리됐는가"
- "테스트가 작성됐는가"

### 머지 전
- "스크린샷 첨부됐는가 (변경 전/후)"
- "regression: 기존 자동 캠페인 분배 정상 작동하는가"
- "PR-3의 경우: staging dry-run 했는가"

## 마이그레이션 안전 규칙 (PR-3)

PR-3 데이터 마이그레이션은 운영 중 시스템에 가장 위험합니다. 시작 전 반드시:

1. Staging 환경 준비 (없으면 PR-3 시작 전 만들기)
2. Prod DB dump → staging에 복원 → alembic 마이그레이션 dry-run
3. 검증 SQL 통과 확인 (PR-3 문서 §마이그레이션 절차 참조)
4. 백업 스냅샷 + 롤백 스크립트 준비
5. 24h 모니터링 일정 잡기 (배포 후 문제 발견 즉시 대응)

## 디자인 원칙 5개 (다시 한번)

CLAUDE.md §4의 5개 원칙은 모든 PR에서 매번 적용:

1. 시각적 위계 = 사용 빈도 + 권한
2. 설정은 즉시 결과 시뮬레이션
3. 자동 작업의 "왜?" 항상 표시
4. 운영자 용어 + 안전 디폴트 + 복원 가능
5. 한 작업의 모든 정보는 한 화면에

이 원칙이 안 지켜진 PR은 머지 전 리뷰에서 다시 작업.

## 변경 추적

이 문서들은 살아있는 문서입니다. 작업 중 가정이 틀린 게 발견되면 해당 PR 문서를 업데이트하고 커밋하세요. 다음 PR이 그 변경을 참조할 수 있도록.

특히 다음을 발견하면 문서 업데이트:
- API schema가 실제로 다르게 가야 함이 드러남
- DB 인덱스 추가 필요
- 엣지 케이스 추가 발견
- 테스트 케이스 추가 필요

---

## 질문이 생기면

각 PR 문서의 §완료 정의가 모호하거나, 충돌하는 결정이 발견되면 작업을 멈추고 운영자(=user)에게 결정 받기. 추측으로 진행하지 말 것.

자주 모호한 영역:
- API 경로 명명 (기존 패턴이 일관되지 않을 때)
- 마이그레이션에서 기존 컬럼 처리 (deprecate vs remove)
- 빈 상태에서 운영자가 다음에 뭘 할지 안내 문구
- AI 응답이 부적절할 때 fallback 동작
